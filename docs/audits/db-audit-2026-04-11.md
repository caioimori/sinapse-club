### Issues

- **[WARN] `user_activity_log` INSERT policy is `WITH CHECK (true)`.** Any authenticated user can insert arbitrary activity rows on behalf of anyone. The trigger functions that write into this table already run as `SECURITY DEFINER`, so this permissive INSERT is probably a leftover. **Fix:** Restrict to `WITH CHECK (auth.uid() = user_id)` OR remove the INSERT policy entirely and rely exclusively on the `SECURITY DEFINER` triggers.
- **[INFO] `free_tier_limits`** has only a SELECT policy. Means inserts/updates can only come from `SECURITY DEFINER` functions (`increment_free_thread_count`), which is fine by design, but should be documented explicitly.

All 15 are necessary and behaviorally safe. ✅

### ⚠️ search_path not pinned (23 functions flagged by security advisor)

Every function in `public` lacks `SET search_path = public, pg_catalog` (or similar). Exploitable only if an attacker can create functions in a schema earlier in the path, which requires existing high privileges. **Low risk in Supabase** (hosted, no untrusted schema creation), but **recommended** to fix for defense-in-depth.

**Fix template:** `ALTER FUNCTION public.fn_name() SET search_path = public, pg_catalog;`

## 7. Advisors Summary

### Security Advisor — 25 findings

| Level | Count | Issue |
|---|---|---|
| WARN | 23 | `function_search_path_mutable` (see §6) |
| WARN | 1 | `rls_policy_always_true` on `user_activity_log` INSERT (see §1) |
| WARN | 1 | `auth_leaked_password_protection` disabled — enable HaveIBeenPwned in Supabase Auth dashboard |

**Top 3:**
1. `user_activity_log` INSERT policy is `WITH CHECK (true)` — fix.
2. Leaked password protection disabled — one-click fix in Supabase dashboard.
3. 23 functions missing `SET search_path` — batch ALTER FUNCTION script.

### Performance Advisor — 125 findings

| Level | Count | Issue |
|---|---|---|
| WARN | 43 | `auth_rls_initplan` — `auth.uid()` re-evaluated per row |
| WARN | 40 | `multiple_permissive_policies` — same cmd has overlapping policies |
| INFO | 31 | `unused_index` — statistics-based, ignore pre-launch |
| INFO | 11 | `unindexed_foreign_keys` — see §3 |

**Top 3:**
1. **`auth_rls_initplan` on 23 tables.** Rewrite policies: `auth.uid() = user_id` → `(select auth.uid()) = user_id`. This lets Postgres cache the value once per query instead of per row. Linear perf win on every authenticated query.
2. **`multiple_permissive_policies` on 8 tables** (badges, benefits, forum_categories, forum_subcategories, lessons, levels, professional_roles, tools). Admin "manage ALL" + "anyone can read" overlap on SELECT for anon. Fix: restrict admin ALL policy to `TO authenticated` only, or scope admin policy to non-SELECT commands.
3. **11 unindexed FKs** — add covering indexes for `posts.quote_of`, `notifications.actor_id` first (highest traffic paths).

## 11. Quick Wins (fastest fixes, highest impact)

1. **Enable Leaked Password Protection** — Supabase Dashboard → Auth → Password Security → toggle on. **Time: 30 seconds.**
2. **Fix `user_activity_log` INSERT policy** — `DROP POLICY "System can insert activity"; CREATE POLICY ... WITH CHECK (auth.uid() = user_id)`. **Time: 2 minutes.**
3. **Add covering indexes for hot FKs** — `posts.quote_of`, `notifications.actor_id`, `posts.last_reply_by`, `marketplace_listings.author_id`, `lesson_progress.course_id`. **Time: 5 minutes.**
4. **Rewrite RLS policies to use `(select auth.uid())`** — largest perf win. Scriptable as a single migration. **Time: 30 minutes.**
5. **Pin `search_path` on 23 functions** — single ALTER FUNCTION migration. **Time: 10 minutes.**

Total: **< 1 hour** to convert CONDITIONAL GO → full GO.
